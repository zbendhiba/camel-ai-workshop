--- title: "Step 04: LLM Classification" layout: :theme/page ---

In the previous step, you sent a single question to an LLM and logged the response. Now you'll use an LLM to make routing decisions. The LLM classifies incoming messages. Camel routes each one to a different handler based on that classification.

This is where Camel's integration patterns and LLMs start working together. A standalone LLM framework can classify text, but it can't route messages across systems. Camel can.

The Content-Based Router

The Content-Based Router is a classic Enterprise Integration Pattern. It inspects each message and sends it to a different destination based on its content. In Camel, this is the choice EIP.

Normally, the router checks a header or a field in the message body. In this step, the "content" is the LLM's classification. The LLM reads customer messages and returns a category: COMPLAINT, QUESTION, or PRAISE. Camel's choice then routes each message to the right handler.

The scenario

Customer messages arrive as text files in a data/ directory. Each message needs to be classified as a COMPLAINT, QUESTION, or PRAISE, then routed to a specialized handler.

Create your Routes

You'll build this step in two parts. Start with the handler routes, already provided. Then create the classification route that reads files, calls the LLM, and dispatches to those handlers.

File (read) --> LLM (classify) --> Choice --> COMPLAINT handler
File (read) --> LLM (classify) --> Choice --> QUESTION handler
File (read) --> LLM (classify) --> Choice --> OTHER handler

Part 1: The handler routes

Open the route.camel.yaml file in your step directory. Three handler routes are already there. Two of them are simple: they consume from a Direct endpoint and log the message.

The direct component is how Camel connects routes together within the same application. When another route sends a message to direct:handle-complaint, it arrives here. Think of it as an internal method call between routes.

- route:
    id: handle-complaint
    from:
      uri: direct:handle-complaint
      steps:
        - log:
            message: "COMPLAINT detected: $\{body}"

- route:
    id: handle-other
    from:
      uri: direct:handle-other
      steps:
        - log:
            message: "OTHER/PRAISE: $\{body}"

The third handler is more interesting. The handle-question route sends the customer message to the LLM to generate an answer. This works exactly like the LLM call you built in step 03.

- route:
    id: handle-question
    from:
      uri: direct:handle-question
      steps:
        - log:
            message: "QUESTION detected: $\{body}"
        - setHeader:
            name: CamelOpenAISystemMessage
            constant: >-
              You are a helpful customer support agent. Answer concisely.
        - to:
            description: Generate answer
            uri: openai:chat-completion
        - log:
            message: "Generated answer: $\{body}"
- route:
    id: handle-question
    from:
      uri: direct:handle-question
      steps:
        - log:
            message: "QUESTION detected: $\{body}"
        - setBody:
            simple: "Answer this customer question concisely: $\{body}"
        - to:
            uri: langchain4j-agent:classifier
        - log:
            message: "Generated answer: $\{body}"

Part 2: The classification route

Now add the main route to the same file. Think of this route as the main agent. It reads each file, asks the LLM to classify it, and decides what to do next. For complaints and praise, it takes a quick action (just logging). For questions, it delegates to the handle-question route. That route calls the LLM again to generate an answer.

This is a common pattern in agentic systems. One agent triages incoming work and dispatches to specialized agents. Here, Camel's choice EIP handles the dispatching.

Add a route named classify-messages to the existing route.camel.yaml file in step-04/langchain4j/. Use the LangChain4j Agent component with agent ID classifier. This route should:

  1. Read files from the ../data directory without moving or deleting them
  2. Save the original message body (you'll need it later)
  3. Send the message to the LLM for classification as COMPLAINT, QUESTION, or PRAISE
  4. Restore the original message body
  5. Use the choice EIP to dispatch to direct:handle-complaint, direct:handle-question, or direct:handle-other

Use the Camel MCP tools and the Kaoto editor to build this route. The Camel MCP Server can help you look up the file component, the LangChain4j Agent component, and the choice EIP.

Note

The data/ directory contains sample messages for each category. You can add your own text files to test the classification.

Solution walkthrough

Warning

The section below describes the solution. Try building the route yourself first using the Camel MCP tools and the Kaoto editor. Come back here once you're done or if you get stuck.

The route has three key parts:

1. Save the original body

In Camel, when you call setBody or send the body to an LLM, the message body is replaced with the new content. The original file content is gone. Use setVariable to store it before the classification so you can restore it later.

- setVariable:
    name: originalMessage
    simple: "$\{body}"

Note

Camel's Simple language is a lightweight scripting language designed for common use cases in Camel routes. It uses placeholders like $\{body} for the message body, $\{headers.name} for headers, and $\{variable.name} for variables.

2. Classify

Embed the classification prompt directly in the body, asking the LLM to reply with just the category name. After the LLM call, the body contains the LLM response. Trim it and save it in a variable.

- setBody:
    simple: >
      Classify the following customer message into exactly one category:
      COMPLAINT, QUESTION, or PRAISE.
      Reply with ONLY the category name, nothing else.

      Message: $\{body}
- to:
    uri: langchain4j-agent:classifier
- setVariable:
    name: category
    simple: "$\{body.trim()}"

3. Restore and dispatch

Set the body back to the original message so the handlers receive the actual customer text, not the classification. Then use choice on the category variable.

- setBody:
    simple: "$\{variable.originalMessage}"
- choice:
    when:
      - simple: "$\{variable.category} == 'COMPLAINT'"
        steps:
          - to:
              uri: direct:handle-complaint
      - simple: "$\{variable.category} == 'QUESTION'"
        steps:
          - to:
              uri: direct:handle-question
    otherwise:
      steps:
        - to:
            uri: direct:handle-other

Use temperature=0.0 for deterministic classification. A higher temperature would introduce randomness that could cause unpredictable routing.

The full solution is in the step-04/langchain4j/solution/ directory.

Going further: structured output with the OpenAI component

The OpenAI component supports a jsonSchema parameter that forces the LLM to return structured JSON instead of plain text. This is more robust than parsing a text response because the schema guarantees the structure.

A classify-schema.json file is already provided in step-04/openai/. It defines a JSON schema with an enum of valid categories:

{
  "type": "object",
  "properties": {
    "category": {
      "type": "string",
      "description": "The message category.",
      "enum": ["COMPLAINT", "QUESTION", "PRAISE"]
    }
  },
  "required": ["category"],
  "additionalProperties": false
}

Try building the same classification route for the OpenAI variant. The handler routes are already in step-04/openai/route.camel.yaml. Add a classify-messages route that uses this schema. Use the Camel MCP tools to look up the OpenAI component, the jsonSchema parameter, and the jsonpath language to extract the category from the JSON response.

Solution walkthrough

Warning

Try building the route yourself first. Come back here once you're done or if you get stuck.

The classification route follows the same structure as the LangChain4j variant, with a few differences.

1. Classify with structured output

The OpenAI component uses the CamelOpenAISystemMessage header for the system prompt. The classification categories go in the system prompt, not in the JSON schema descriptions. Small local models like Ollama don't reliably read those. The jsonSchema parameter points to the schema file, and temperature=0.0 keeps the output deterministic.

- setHeader:
    name: CamelOpenAISystemMessage
    constant: >-
      You are a message classifier. Classify the customer message
      into exactly one category: COMPLAINT, QUESTION, or PRAISE.
- setBody:
    simple: "Classify this message:\n$\{body}"
- to:
    uri: openai:chat-completion
    parameters:
      jsonSchema: "resource:classpath:classify-schema.json"
      temperature: 0.0

2. Extract the category with jsonpath

The LLM returns JSON like {"category": "COMPLAINT"}. Use jsonpath to extract the category into a variable instead of body.trim().

- setVariable:
    jsonpath:
      expression: $.category
    name: category

3. Restore and dispatch

This part is the same as the LangChain4j variant. Restore the original message body and use choice on the category variable.

- setBody:
    simple: "$\{variable.originalMessage}"
- choice:
    when:
      - simple: "$\{variable.category} == 'COMPLAINT'"
        steps:
          - to:
              uri: direct:handle-complaint
      - simple: "$\{variable.category} == 'QUESTION'"
        steps:
          - to:
              uri: direct:handle-question
    otherwise:
      steps:
        - to:
            uri: direct:handle-other

The full solution is in step-04/openai/solution/.
